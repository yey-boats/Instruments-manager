Manager-embedded visual editor for MIDL tile layouts — goals, panes, flow, and gap analysis

Manager App — Embedded Editor v1 Scope MIDL Config Authoring SignalK Binding
01 — Goals

What the Builder Must Achieve

⬡
Author MIDL tile configurations visually inside the Manager app — no hand-editing of JSON files required.
⬡
Bind SignalK paths to tiles via a live data tree — point-and-click path selection with live value preview.
⬡
Validate authored configs against the device manifest schema before any push — structural, semantic, and capability checks.
⬡
Push the validated MIDL config directly to a selected device — the device pulls and re-renders immediately.
02 — Authoring Model

Generic-Author, Device-at-Push

Dashboards are authored generically — not device-first. The Builder works with a MIDL config that describes tile types, bindings, layout, and properties without being tied to a specific physical unit.

A device is selected only as the push destination. When you click "Push to device", you choose which connected instrument receives the config. The device class drives the manifest, which constrains which tile types are structurally valid and what properties each tile may carry.

The device renders whatever valid MIDL it receives — the Builder's job is to guarantee that validity before delivery. This separation keeps authored configs portable across multiple units of the same class.

Key Invariant

The Builder never stores a device serial number inside a config file. The config describes layout + tile-type + signalK-path + display-props. The physical target is supplied only at push time.

Manifest/schema is loaded by device class (e.g. esp32-4848s040 · square-480), not by serial. Multiple devices of the same class share one schema.

03 — V1 Panes

Editor Layout — Three Panes

Left Rail
210 px
Elements Data

Elements tab — palette of tile types sourced from the loaded manifest. Drag or click-to-insert.

  • Single-Value basic readout
  • Hero large primary value
  • Compass bearing ring
  • Wind Rose apparent / true wind
  • Gauge / Bar range indicator
  • Trend sparkline history
  • Autopilot AP control tile
  • Button momentary action

Data tab — live SignalK path tree with streaming values. Click a path to bind it to the selected tile.

Center Canvas
flex

Device frame rendered at true resolution with a visual grid overlay. Selection handles appear on the active tile.

  • Device frame at true px resolution 480×480 for square-480
  • Grid cells aligned to tile budget
  • Selection handles on active cell
  • Tile budget indicator 480×480 · 3/4 tiles
  • Add cell / remove cell controls
  • SVG live preview rendered by midl renderer
V1 constraint

Tile placement via grid cells only. Free drag-drop canvas is deferred to Full.

Right Inspector
220 px

Selected tile's MIDL properties. All fields map 1-to-1 to MIDL schema properties.

  • Type tile type selector
  • Binding vessels.self.navigation.speedOverGround
  • Live value preview current SignalK value
  • Unit e.g. kn, °, m/s
  • Decimals display precision
  • Span tile grid span 1–4
  • Sided port / stbd / both
  • Color theme override or auto

05 — Validation + Push Flow

From Authoring to Device Render

1
Author tiles in Visual mode or Source mode
Place tiles on the grid canvas, set type and SignalK binding via the Inspector, or edit MIDL text directly in Source mode. Both modes stay in sync.
2
Validation runs continuously Live
Three layers checked on every edit: structural (valid MIDL schema), semantic (paths exist in current SignalK stream), capability (tile type is supported by the selected device class manifest). Errors surface inline.
3
Select push destination Target
Click the "Push to device ▾" dropdown to choose a connected device by serial number and friendly name. Only online devices are listed.
4
Confirm diff and push Diff
A summary dialog shows the change count, e.g. "3 tiles change, 1 tile added". Confirm to transmit the validated MIDL payload to the device.
5
Device pulls new config and re-renders
The firmware receives the new MIDL config, parses it, and re-renders the display using the existing tile renderer pipeline. No firmware update required.
✓
Status confirmed Pushed
Manager receives acknowledgement from the device. Status banner shows: ✓ Pushed — device pulled config. Autosave records the pushed config locally.
